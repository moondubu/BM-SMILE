# BM-SMILE 프로젝트 리뷰/회고

## 1) 잘한 점
- 피드백-수정-푸시 사이클이 빨라 디자인 싱크를 짧은 루프로 맞출 수 있었다.
- 공통 컴포넌트(`SlideCard`, `Hero`, 섹션 컴포넌트) 기반으로 후반 수정 속도가 빨라졌다.
- 모바일/태블릿/PC를 분리해 조정하며 breakpoint 기준을 점진적으로 안정화했다.
- 배포 이슈(favicon/manifest/og:image 경로)도 원인 추적 후 재현 가능한 형태로 정리했다.

## 2) 실수/비효율
- “모바일만 변경” 요청인데 PC까지 같이 변경되는 경우가 반복됐다.
- 공통 컴포넌트 수정과 페이지 오버라이드 경계를 초반에 명확히 잡지 못해 중복 CSS와 되돌리기가 발생했다.
- 토큰 사용 vs 하드코딩 요구를 즉시 확정하지 못해 재수정이 생겼다.
- 반응형 작업 중 구조 변경이 들어가 의도(구조 유지 + 노출 제어)와 어긋난 경우가 있었다.
- GitHub Pages `basePath` 환경 검증을 초기에 못 해서 후반 배포 수정이 발생했다.

## 3) 효율적이었던 방식
- “해당 셀렉터만 오버라이드” 방식으로 영향 범위를 좁혀 수정한 점.
- 공통 규칙으로 이동시키고 페이지별 중복을 제거하는 정리 방식.
- 이슈 발생 시 컴포넌트 CSS / 페이지 CSS / 배포 설정으로 원인 범위를 분리해 빠르게 진단한 점.
- 작은 단위 커밋으로 QA 확인 사이클을 빠르게 유지한 점.

## 4) 스킬/룰로 저장할 항목
### A. 반응형 수정 룰
- 요청된 breakpoint 외 수정 금지.
- 모바일 요청 시 PC diff 확인.
- 구조 변경 금지, 우선 `display/order/spacing`으로 해결.

### B. 공통/오버라이드 룰
- 공통 컴포넌트 수정 전 사용 페이지 확인.
- 페이지 오버라이드는 페이지 prefix 강제.
- 중복 선언 발견 시 공통/개별 중 한쪽으로 정리.

### C. 토큰/픽셀 적용 룰
- 요청을 먼저 분류: “토큰 대응” vs “고정 px”.
- 고정 px 요청이면 remap 토큰 사용 금지.
- 머지 전 토큰/하드코딩 혼재 점검.

### D. 배포 메타 룰
- `basePath/assetPrefix` 환경에서 favicon/manifest/og:image URL 점검.
- production 렌더 결과의 `<meta property="og:image">` 실제 값 확인.
- 로컬/배포 URL 불일치 체크리스트 고정.

### E. 푸시 전 QA 룰
- 변경 페이지 PC/Tablet/Mobile 3구간 확인.
- 요청 범위 외 변경 여부 확인.
- 콘솔 경고/에러(passive, preload, touch) 확인.

## 5) 다음 프로젝트 개선안
- 작업 시작 시 1줄 합의: 대상 페이지 / breakpoint / 구조 변경 여부 / 토큰 여부.
- 각 피드백 반영 후 영향 범위 3개(공통컴포넌트, 페이지 오버라이드, 배포 설정) 점검.
- 반복되는 아이콘/이미지 전환은 variant prop으로 모델링해 재작업 비용 절감.
- 릴리즈 전 통합 QA 패스를 고정 프로세스로 운영.
